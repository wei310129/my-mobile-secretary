# 對話問題與情境清單開發進度

最後更新：2026-07-19（Asia/Taipei）

## 強制開發順序

1. 每次開始下一項功能前，先查 `intent_issue` 的新 `OPEN` 回報。
2. 新回報優先於評估清單與情境清單；完成並回歸後才更新 checkpoint。
3. 對話紀錄評估的缺口全部完成後，才從情境清單第一個「部分開發／未開發」項目依序往下。
4. 每項至少要有服務層安全規則與回歸測試；只有提示詞或安全拒絕仍算部分開發。

目前問題回報 checkpoint：`intent_issue.id = 132`。#125–#132 已完成；963 項完整回歸通過。
依使用者指示尚不進入情境清單；再次開發前查 `id > 132` 的新 OPEN 回報。

## 已完成的對話問題

- 同名待辦／行程修改先消歧，不直接改錯資料。
- 老師通知缺結束時間時保留草稿並追問；多日期、多主題通知逐項拆解。
- 孩子送課後不假設由使用者接；接送人、時間、地點不完整時主動確認。
- 意圖提示上下文包含行程結束、週期、截止與地點。
- 模糊的用餐、工作與睡眠時間不硬猜；唯讀查詢不被誤攔。
- 日期範圍會真正限制工作量查詢，不只寫在提示詞。
- 家庭成員、學校與老師通知可使用已確認的個人知識補值。
- 今天日期、民國年、星期矛盾、年份歧義與穩定日期回顯。
- 行程地點回答會檢查實際保存的 `placeId`。
- 活動公告文字／海報先建立私人草稿；購票後逐步確認時間、地點、提早到、提醒與準備事項，最後明確確認才建行程。
- LINE 不宣稱取得即時 GPS；App 只採信 30 分鐘內定位，天氣可由附近已知地點推導鄉鎮，否則清楚降級到縣市。
- 平日／假日早餐、午餐、晚餐、睡眠保存為生活時間窗，不偽裝成固定行程；可行性引擎會提示時段被壓縮並讓使用者決定，支援跨午夜睡眠。
- 國定假日／停班停課條件式週期使用獨立規則與執行紀錄；資料未知時停在等待狀態，不虛構官方結論。規則引擎與對話流程已完成，正式環境仍需設定官方政府日曆 gateway。
- 農曆日期支援年份確認、國曆轉換、閏月消歧與年度預覽；設定 CWA API key 後使用中央氣象署官方農民曆資料，轉換失敗時不建立行程。
- #152 家庭接送可在一句話中建立兩筆時間點行程，保留老婆／阿公責任人與目的地，不算成使用者本人忙碌，也不虛構活動時長。
- #153 同句連續改口只採最後一次改期時間；以原日期與鐘點尋找唯一既有行程，保留原時長與地點，不新增資料；多候選與固定週期先澄清。
- #154 可把明日明確時間窗與所需連續時長傳給空檔引擎，最多回覆使用者要求的選項數且全程唯讀；其他家人負責的家庭事件不再占用本人空檔。

## 驗證注意事項

- 多條開發線共用同一個 workspace 與 `target`，不可同時執行 Maven lifecycle；由主線統一編譯與測試。
- 根專案 Maven lifecycle 統一優先使用 `scripts/mvn-safe.ps1`：跨程序互斥且預設不 clean；只有
  已確認舊 class／產生碼污染或正式完整驗收才明確使用 `-Clean`，避免刪除 `target` 評估輸出。
- Maven 在受限 Windows 沙箱內可能出現 `Fatal Error: Cannot close compiler resources`；同一命令改在
  工作區沙箱外可穩定通過，並非原始碼編譯錯誤。
- migration／RLS 需使用 Docker/Testcontainers 執行 `WorkspaceRlsIntegrationTest`。
- `.claude/`、`.codex/` 與其他未追蹤內容屬於使用者，不能刪除或覆蓋。
